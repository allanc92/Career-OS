# Applications

Create one folder per opportunity by copying
`_TEMPLATE_Org_YYYY-MM-DD` to `Org_YYYY-MM-DD`. Use the date work on the
application began. Replace spaces and unsafe filename characters with
underscores. Keep the stable Opportunity ID in every related file.

Real application folders are personal and should remain untracked. The tracked
template contains no personal data.

## Standard files

- `ApplicationNotes.md`: role facts, fit evidence, gaps, claims, materials,
  tailoring decisions, and next action
- `ConversationLog.md`: the user's human-authored contact and conversation notes

Additional role descriptions, prep documents, and approved application
materials may live in the same folder. Use clear filenames and dates.

## Statuses

Application status must match the newest `Pipeline/Activity_Log.md` row for the
same Opportunity ID. Use only: `Lead`, `Researching`, `Networking`, `Preparing`,
`Applied`, `Recruiter Screen`, `Interviewing`, `Final Round`, `Offer`,
`Negotiating`, `Accepted`, `Rejected`, `Withdrawn`, `Closed`, or `Paused`.

## Grounding and approval

- Start from real sources: the role description, master resume, Achievement Log,
  StoryBank, and user-provided notes.
- Put `[TBD]` beside missing facts. Never create a claim, metric,
  motivation, relationship, or qualification.
- Show the proposed folder name and file contents before creating them. A direct
  instruction to create or save a named application counts as approval only for
  the displayed scope.
- Show every proposed status row before appending it unless the user directly
  requested that exact update.
- Ask the one-time privacy question before the first real save. Real details
  should stay local or in the user's own private repository.
